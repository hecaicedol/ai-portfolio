02 · The differentiator

Why hire a mathematician for AI engineering?

The honest answer — and the three things mathematics gives me that a CS degree alone wouldn't.

"AI engineering is hard because correctness is hard."

Mathematics is the only discipline that spends years teaching you how to prove what you claim — and how to know exactly when your proof breaks. Production AI fails the same way undergraduate proofs fail: hidden assumptions, untested cases, ambiguous definitions, missing edge cases.

The difference is that in production, no one is grading you. The system just quietly produces a wrong answer — and someone pays for it. That's the gap this portfolio is built to close.

↳ Anthropic, DeepMind, OpenAI hire heavily from math and physics. Not by accident.

02c · Mathematics in motion · Reciprocal Rank Fusion

Two parameter-free lines of math beat a model that costs a million dollars to train

P6 fuses keyword (BM25) and vector (cosine) retrieval with Reciprocal Rank Fusion. One constant k = 60 (recommended in the 2009 paper and never seriously challenged), no training, no parameters to tune. It beats both signals alone on every published benchmark. Try it across four different corpora, toggle the retrieval mode, change k, then hit the LLM-synthesis button to see what a downstream model would actually return.

1. pick a corpus → see how vector vs keyword split on real data 2. slide k and switch between Vector / Keyword / Hybrid modes 3. hit 🤖 Synthesize → the top-3 contexts go into a streaming "LLM" answer
score(d) = Σi  1 / (k + ranki(d))   ·   k = 60
A doc ranks higher when it sits near the top of multiple lists. The reciprocal damps any single signal — no list can completely overrule another. Bigger k = more "patience" (a doc at rank 20 still contributes meaningfully); smaller k = aggressive top-of-list weighting.

        02d · Mathematics in motion · Self-attention

        The 49-character formula behind every model in this portfolio

        Every LLM you've heard of — Claude, GPT, Llama — is built on stacked layers of one tiny operation: scaled dot-product attention. Below, hover any output token to see which input tokens contributed to it and by how much. The line weight is literally the attention weight after softmax. Click shuffle for a fresh pattern.

        1. hover any token in the bottom (output) row 2. the cubic-bezier edges show that token's attention distribution after softmax 3. press ↻ Shuffle attention to regenerate the weights from a fresh kernel
        Attention(Q, K, V) = softmax(QKT / √dk) · V
        Q, K, V are linear projections of the input (queries, keys, values). The dot product QKT measures alignment; dividing by √dk keeps the variance ≈ 1 so softmax doesn't saturate as the embedding dimension grows; the softmax row turns alignments into a probability distribution that weights V. The whole portfolio is layers of this on top of each other.
        hover an output token →
        Hover a token in the bottom row to see its attention distribution.

        What you're seeing. The bottom row is the model "generating" each output token; the top row is what it "looked at". For a translation task, you'd expect the model to attend to a roughly diagonal pattern (each output token aligns with one or two input tokens) — but for actual language, attention is often surprisingly distributed: pronouns attend to their antecedents many tokens away, verbs attend to subjects, articles barely attend to anything. This is why attention is the mechanism that lets a model handle long-range dependencies without a recurrent state.

        Why the √dk matters. Without it, as you scale embedding dimension dk from 64 to 1024, the variance of Q·K grows linearly, the softmax saturates (one weight becomes ≈1, all others ≈0), and gradients vanish. The √dk normalization preserves a healthy gradient signal across model scales. It's a one-character fix, and it's why the 2017 "Attention is All You Need" paper works at all.

        PROJECT 01 / 06 ● Functional

        Self-Healing Multi-Agent Pipeline

        Constitutional AI critic · Bounded reflection · pgvector episodic memory

        The problem

        In production, the most expensive AI failures are silent: a confident wrong invoice number propagates, a downstream system trusts the bad output, the wrong vendor gets paid. Standard pipelines treat the LLM as a black box — if it returns something, the pipeline returns "success."

        This pipeline does the opposite: every output is adversarially evaluated by a critic, failures trigger a bounded reflection loop, and the system maintains an episodic memory of past errors that the critic consults before every evaluation.

        Key architectural decisions
        LangGraph orchestrationExplicit, inspectable state transitions over LangChain's opaque chains
        Constitutional AI criticPer-principle scoring (4 axes) yields actionable feedback; single-rubric LLM-as-judge does not
        Episodic memory in pgvectorDomain-specific failure patterns surface across thousands of past runs — reuses the Postgres we already need
        Hard cap: 3 reflection iterationsUnbounded retry loops are a top-3 incident class for agents. Cap forces loud failures over silent token spend
        Lessons learned the hard way
        • Critics need memory, not just rubrics. A stateless critic re-makes the same mistake every run.
        • Constitutional principles must be evaluable, not philosophical. "Output should be helpful" is unscorable; "every required field has a non-empty value matching its type" is.
        • Reflection loops without hard caps burned $40 of credits on a single malformed PDF in week one.
        Implementation status
        • Backend: 5-node LangGraph (extract → validate → critique → reflect → synthesize), 4 agents implemented, FastAPI with SSE streaming, pgvector episodic memory layer.
        • Tests: 22 pytest tests passing in ~3s — validator, agent JSON-retry, all four orchestrator paths, FastAPI HTTP layer. Zero Docker, zero API-key, zero cost to run.
        • Evaluation harness: 15-document synthetic corpus (8 invoices · 4 receipts · 3 contracts, easy/medium/hard mix) plus a runner that grades extracted output field-by-field against ground truth and emits aggregate metrics.
        • Frontend: Next.js 14 + Tailwind SPA with live SSE pipeline view, per-principle critic scoring, episodic memory panel. Includes a client-side scripted demo mode that plays the self-healing scenario without any backend.
        • Pending (visible in the repo): populate the metrics card with the real benchmark numbers, public Vercel deployment, Langfuse callback wiring.

        PROJECT 03 / 06 ● Core memory verified

        Autonomous Research Agent · MemGPT pattern

        Three-tier memory · Persistent state · Tavily + arXiv + PDF reports

        The problem

        A research agent that "forgets" between sessions is a chatbot, not a researcher. The MemGPT paper showed that explicit memory tiering — working / episodic / semantic — with a controller that promotes and evicts beats relying on bigger context windows.

        Key architectural decisions
        Three-tier memoryWorking (in-context) / episodic (recent runs) / semantic (consolidated facts). Each tier has explicit promotion rules.
        Memory controller as a nodePromotion and eviction are first-class graph nodes — inspectable in Langfuse, not buried in a hook
        arXiv + Tavily + scrapingMix of structured (arXiv API) and unstructured (Tavily) sources reduces source-bias in synthesized reports
        Implementation status
        • Implemented + tested: WorkingMemory (token-budget FIFO), InMemoryEpisodic (cosine-similarity recall + per-session archive), InMemorySemantic (cosine dedupe at 0.92 + source-union on collision), MemGPTController (auto-archives evictions, assembles working+episodic+semantic context, consolidates at session end), and the LangGraph research agent (planner→executor→reflector→reporter) driven by a ScriptedLLM. 31 pytest tests passing in ~1 s, zero Docker, zero API keys.
        • Scaffolded: PostgresEpisodic / PostgresSemantic (pgvector backends — docker-compose service is already declared), Next.js memory viewer, real run against live Tavily + arXiv + Claude.
